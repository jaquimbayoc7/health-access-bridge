1. Contexto y Tarea Asignada

Origen de la tarea

Tras completar el proceso de investigación HCI (entrevistas semi-estructuradas, user personas, user story mapping y tests de usuario), se identificaron hallazgos críticos de usabilidad en el frontend de Health Access Bridge. El equipo evaluó dos planes de mejora y seleccionó la Opción B — Plan completo.

Hallazgos HCI que motivaron la tarea

  • H1 — Terminología ICF opaca: Los campos D1–D6 no tenían contexto ni ejemplos. Los médicos desconocían la escala 0–100.
  • H2 — Confirmación de borrado genérica: El diálogo de eliminación de paciente no mostraba el nombre del afectado, generando ansiedad y riesgo de error.
  • H3 — Errores API sin mensajes claros: Los errores 401/422/500 producían pantallas en blanco o mensajes en inglés sin orientación al usuario.
  • H4 — Sin onboarding: Usuarios nuevos debían explorar el sistema sin ninguna guía inicial.
  • H5 — Sin Centro de Ayuda: No existía documentación in-app para la terminología ICF ni las funciones del sistema.
  • H6 — Sin indicación de expiración de sesión: El JWT expiraba sin aviso, causando pérdida de datos del formulario.
  • H7 — Sin breadcrumbs de navegación: Los usuarios perdían el contexto de ubicación dentro de la aplicación.

S1-1 — Tooltips contextuales para campos ICF (D1–D6)

S1-2 — Confirmación de eliminación contextual
frontend/src/pages/Patients.tsx · MODIFICADO

Problema HCI: El diálogo de confirmación mostraba un texto genérico ("¿Está seguro?") sin identificar al paciente. Los médicos expresaron inseguridad sobre si estaban eliminando al paciente correcto.

Solución implementada: El AlertDialog ahora muestra un bloque destacado con el nombre completo y número de documento del paciente que será eliminado, siguiendo el principio heurístico de "reconocimiento en lugar de recuerdo".

S1-3 — Manejo centralizado de errores API en español
frontend/src/services/api.ts · MODIFICADO  |  frontend/src/services/patients.ts · MODIFICADO

Problema HCI: Los errores de red mostraban mensajes en inglés o pantallas en blanco. Los médicos no sabían si el error era transitorio o requería acción.

Solución implementada: Método privado handleResponse<T> en ambos servicios (ApiService y PatientService) con mensajes en español para cada código HTTP. Se eliminaron 3 usos de window.location.href = '/login' reemplazados por el callback centralizado ApiService.onUnauthorized.

switch (response.status) { case 401: ApiService.onUnauthorized?.(); throw new Error('Sesión expirada. Por favor, inicie sesión nuevamente.'); case 403: throw new Error('No tiene permisos para realizar esta acción.'); case 422: throw new Error('Datos inválidos. Verifique la información ingresada.'); case 500: throw new Error('Error temporal del servidor. Intente de nuevo.'); }

5. Sprint 3 — Accesibilidad y Gestión de Sesión

⏱️
S3-2 — Hook useJWTExpiry + logout automático en AuthContext
frontend/src/hooks/useJWTExpiry.ts · NUEVO  |  frontend/src/contexts/AuthContext.tsx · MODIFICADO

Problema HCI: El JWT expiraba silenciosamente. Al intentar guardar un formulario largo, la sesión estaba inválida y se perdían todos los datos ingresados.

Solución implementada: Hook useJWTExpiry que:

  1. Parsea el payload del JWT en localStorage para obtener el campo exp.
  2. 5 minutos antes de expirar: lanza un toast.warning con cuenta regresiva visual.
  3. Al expirar: lanza un toast.error y ejecuta el callback logout() automáticamente.

En AuthContext se registró el callback ApiService.onUnauthorized para que tanto el hook como el interceptor API disparen el mismo flujo de logout.

🟢 Sesión activa ⚠️ Aviso 5 min 🔴 Sesión expirada
t=0 t=exp-5min → toast.warning() t=exp → logout()
// useJWTExpiry.ts — lógica principal const decoded = JSON.parse(atob(token.split('.')[1])); const expiresAt = decoded.exp * 1000; const msUntilExpiry = expiresAt - Date.now(); const warnAt = msUntilExpiry - 5 * 60 * 1000; // 5 min antes toast.warning('Tu sesión expirará en 5 minutos...'); onLogout(); // callback de AuthContext

S3-1 — Accesibilidad (aria) + descripción del formulario ICF
frontend/src/pages/Patients.tsx · MODIFICADO

Problema HCI: El formulario de pacientes carecía de atributos de accesibilidad. Los campos ICF no tenían descripción semántica para lectores de pantalla.

Cambios aplicados en el formulario de pacientes:

  • aria-label="Formulario de paciente" en el elemento <form>.
  • aria-describedby="icf-scale-hint" en los 6 campos D1–D6, con nodo sr-only que dice "Escala ICF: 0 sin barrera, 100 barrera completa".
  • aria-describedby="nivel-global-hint" en el campo nivel global, con texto visible "Calculado automáticamente como promedio de D1–D6".
  • Descripción del diálogo actualizada a español con contexto sobre la escala ICF.
<!-- Hint semántico para lectores de pantalla --> <p id="icf-scale-hint" className="sr-only"> Escala ICF: 0 sin barrera, 100 barrera completa </p> <!-- Hint visible para nivel global --> <p id="nivel-global-hint" className="text-xs text-muted-foreground"> Calculado automáticamente como promedio de D1–D6. </p>
